# Init Skill Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement a role-based `init` skill that initializes `.business-builder/` when missing, captures business state (`novo|existente|em_construcao`), and orchestrates validation handoff to `product-owner` before downstream artifacts.

**Architecture:** Build `skills/init/SKILL.md` as an orchestration skill (not a final validator), then add reusable workspace templates under `skills/init/templates/.business-builder/` for deterministic file generation. `init` must guarantee `.business-builder/` exists, ask business state explicitly, create-or-update `idea.md` and `users.md` for `existente`/`em_construcao`, and route for `product-owner` validation before continuation. Validate behavior with RED/GREEN/REFACTOR pressure scenarios and record evidence in `skills/init/validation/test-log.md`.

**Tech Stack:** Markdown (GitHub-flavored), git, grep/awk shell checks, Copilot subagent runs for pressure-scenario simulation.

- [ ] **Step 1: Create `skills/init/SKILL.md` with deterministic orchestration contract**

```markdown
---
name: init
description: Use when starting a new business idea and you need structured scope intake, state-aware artifact bootstrap, and handoff to product-owner validation before downstream skills.
---

# Init

## Skill Type
Technique

## Required Background / Required Sub-Skill
- **REQUIRED BACKGROUND:** None.
- **REQUIRED SUB-SKILL:** None.

## Overview
This skill initializes the `.business-builder` workspace and structures idea scope. It validates business stage context and routes to `product-owner` for approval before downstream flow.

## When to Use
- New business idea has not been structured yet.
- Scope inputs are scattered and inconsistent.
- Team needs a deterministic starting point before deeper analysis.
- Keywords people will search: init scope, business idea intake, startup brief, pre-product handoff.

## When NOT to Use
- Detailed pre-product management analysis (use `product-management`).
- Final RF/RNF/backlog completion for developers (use output skill).

## Quick Reference
| Situation | Action | Why |
|---|---|---|
| Idea is broad/vague | Run intake questions and create/update `idea.md` | Creates a stable baseline |
| Business is existing or in-progress | Create/update `idea.md` and `users.md` | Preserves current context before expansion |
| Finance scope missing | Return `mandatory_adjustments` with missing fields | Prevents fake assumptions |
| Intake is complete and structured | Route package to `product-owner` | Ensures quality gate before continuation |

## Implementation
### Core Pattern
1. Collect mandatory intake fields: idea, business_state (`novo|existente|em_construcao`), objective, audience, constraints, stage, assumptions.
2. Ensure `.business-builder/` exists before writing artifacts.
3. Create/update baseline artifacts (`idea.md` and `users.md`) when business_state is `existente` or `em_construcao`.
4. Write structured context artifacts (`context/brief.md` and `context/decisions.md`).
5. Initialize management and output folder placeholders.
6. Build handoff package for `product-owner`.
7. Return deterministic status and missing-field list when needed.

### Mandatory skill output
| Section | Mandatory content |
|---|---|
| Final status | `approved`, `mandatory_adjustments`, `gate_failed` |
| Intake summary | 3-5 lines of normalized business idea context |
| Missing fields | Explicit list or `none` |
| Artifacts initialized | Paths created/updated |
| Next handoff | Target skill + expected deliverables |

### Gate rules
1. If `business_state`, objective, or audience is missing, status = `mandatory_adjustments`.
2. If business_state is `existente` or `em_construcao`, `idea.md` and `users.md` must be created if missing and updated if present.
3. Init routes validation to `product-owner`; it does not finalize business viability decisions.
4. If required artifacts and handoff are complete, status = `approved`.

## Common Mistakes
| Mistake | Why it fails | Fix |
|---|---|---|
| Generating RF/RNF directly in init | Breaks skill boundaries | Route to output skill after management stage |
| Inventing financial assumptions | Creates false certainty | Mark missing fields and request clarification |
| Skipping business state classification | Breaks deterministic routing | Require `business_state` at intake |
| Not updating existing artifacts | Loses business context continuity | Enforce create-or-update for `idea.md` and `users.md` |
| Deciding viability in init | Violates ownership model | Route validation and readiness check to `product-owner` |

## Rationalization Table (Excuse vs Reality)
| Excuse | Reality |
|---|---|
| "Let's skip intake because deadline is tight." | Skipping intake causes wrong downstream artifacts. |
| "We can invent budget now and adjust later." | Invented finance data corrupts management decisions. |
| "We don't need to ask business state; it's obvious." | State drives deterministic routing and artifact behavior. |
| "Init can just decide viability quickly." | Viability/readiness gate is validated via `product-owner`, not decided ad hoc by init. |

## Red Flags - STOP
- "Generate final backlog now."
- "Assume missing finance values."
- "Skip business state question."
- "Ignore existing `idea.md`/`users.md`."
- "Decide viability in init."

If any red flag appears, restart from Core Pattern step 1.

## TDD for Skills (mandatory)
### RED
- Capture failures without this skill in `skills/init/validation/test-log.md`.

### GREEN
- Re-run same scenarios with this skill loaded and confirm compliant choices.

### REFACTOR
- Add new rationalizations and counters, then retest.
```
