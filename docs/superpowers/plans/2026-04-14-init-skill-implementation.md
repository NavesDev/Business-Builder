> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement a role-based `init` skill that captures the core business idea scope and orchestrates structured handoff into `business-builder/context`, `business-builder/product-management`, and `business-builder/product-output`.

**Architecture:** Build `skills/init/SKILL.md` as an orchestration skill (not a final validator), then add reusable workspace templates under `skills/init/templates/business-builder/` for deterministic file generation. Validate behavior with RED/GREEN/REFACTOR pressure scenarios and record evidence in `skills/init/validation/test-log.md`.

**Tech Stack:** Markdown (GitHub-flavored), git, grep/awk shell checks, Copilot subagent runs for pressure-scenario simulation.

- [ ] **Step 1: Create `skills/init/SKILL.md` with deterministic orchestration contract**

```markdown
---
name: init
description: Use when starting a new business idea and you need structured scope intake, context artifacts, and handoff routing before product-management and final output skills.
---

# Init

## Skill Type
Technique

## Required Background / Required Sub-Skill
- **REQUIRED BACKGROUND:** None.
- **REQUIRED SUB-SKILL:** None.

## Overview
This skill initializes the business-builder workspace and structures idea scope. It does not decide final business viability and does not finalize dev handoff output.

## When to Use
- New business idea has not been structured yet.
- Scope inputs are scattered and inconsistent.
- Team needs a deterministic starting point before deeper analysis.
- Keywords people will search: init scope, business idea intake, startup brief, pre-product handoff.

## When NOT to Use
- Detailed pre-product management analysis (use `product-management`).
- Final RF/RNF/backlog completion for developers (use output skill).

## Quick Reference
| Situation | Action | Why |
|---|---|---|
| Idea is broad/vague | Run intake questions and create `context/brief.md` | Creates a stable baseline |
| Finance scope missing | Return `mandatory_adjustments` with missing fields | Prevents fake assumptions |
| User asks init for viability verdict | Route to `product-management` | Preserves responsibility boundaries |

## Implementation
### Core Pattern
1. Collect mandatory intake fields: idea, objective, audience, constraints, stage, assumptions.
2. Write structured context artifacts (`context/brief.md` and `context/decisions.md`).
3. Initialize management and output folder placeholders.
4. Build handoff package for `product-management`.
5. Return deterministic status and missing-field list when needed.

### Mandatory skill output
| Section | Mandatory content |
|---|---|
| Final status | `approved`, `mandatory_adjustments`, `gate_failed` |
| Intake summary | 3-5 lines of normalized business idea context |
| Missing fields | Explicit list or `none` |
| Artifacts initialized | Paths created/updated |
| Next handoff | Target skill + expected deliverables |

### Gate rules
1. If objective or audience is missing, status = `mandatory_adjustments`.
2. If user requests viability verdict, init routes to `product-management` and does not decide go/no-go.
3. If context artifacts and handoff are complete, status = `approved`.

## Common Mistakes
| Mistake | Why it fails | Fix |
|---|---|---|
| Generating RF/RNF directly in init | Breaks skill boundaries | Route to output skill after management stage |
| Inventing financial assumptions | Creates false certainty | Mark missing fields and request clarification |
| Deciding viability in init | Violates ownership model | Route viability analysis to `product-management` |

## Rationalization Table (Excuse vs Reality)
| Excuse | Reality |
|---|---|
| "Let's skip intake because deadline is tight." | Skipping intake causes wrong downstream artifacts. |
| "We can invent budget now and adjust later." | Invented finance data corrupts management decisions. |
| "Init can just decide viability quickly." | Viability belongs to `product-management`, not init. |

## Red Flags - STOP
- "Generate final backlog now."
- "Assume missing finance values."
- "Decide viability in init."

If any red flag appears, restart from Core Pattern step 1.

## TDD for Skills (mandatory)
### RED
- Capture failures without this skill in `skills/init/validation/test-log.md`.

### GREEN
- Re-run same scenarios with this skill loaded and confirm compliant choices.

### REFACTOR
- Add new rationalizations and counters, then retest.
```
